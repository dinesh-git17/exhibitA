## S2 Protocol Zero

**Priority: HIGHEST. This section supersedes all other output rules.**

Protocol Zero is the attribution erasure standard for this repository. All code, comments, commit messages, and
documentation produced by AI agents MUST be indistinguishable from work written by a senior engineer.

### 2.1 Absolute Prohibitions

The following are forbidden in all source files, comments, commit messages, PR descriptions, and documentation:

- "Generated by **any AI name**" or equivalent attribution.
- "AI-assisted", "AI-generated", "created with AI", "written by AI".
- `Co-authored-by` trailers referencing any AI system or vendor.
- Conversational artifacts: "I hope this helps", "As requested", "Here is the code", "As an AI".
- Hedging language: "I think maybe", "This should work", "You might want to".
- Apologetic preambles, motivational filler, or tutorial-style narration.
- `// TODO: implement`, placeholder stubs, or scaffolding comments.

### 2.2 Enforcement

The repository includes `scripts/protocol-zero.sh`, an automated scanner that detects prohibited attribution patterns
across the codebase and in commit messages. This script MUST pass with exit code 0 before any work is declared complete.

```bash
./scripts/protocol-zero.sh
```

A non-zero exit code is a blocking violation. The agent MUST remediate all findings before proceeding.

### 2.3 Typographic Lint

Typographic characters from rich text editors and AI output are prohibited in all source files. Em dashes (U+2014), en
dashes (U+2013), and smart quotes (U+201C/D, U+2018/9) are forbidden in code.

```bash
./scripts/check-em-dashes.sh
```

A non-zero exit code is a blocking violation. Replace typographic characters with ASCII equivalents before proceeding.

### 2.4 Rationale

Code in this repository has one author: the repository owner. AI agents are tools, not collaborators. Attribution of
authorship to an AI system is a governance violation regardless of context.

## S3 Code & Comment Standards

### 3.1 Voice

All code MUST read as if written by a senior engineer with deep platform expertise. No tutorial commentary. No defensive
verbosity. No filler.

### 3.2 Comments

- Comments exist only where logic is non-obvious or architectural intent requires explanation.
- Never restate what the code already expresses.
- Doc comments on public interfaces only.
- No inline narration explaining "what this does" when the function name and types already communicate it.
- Comments MUST be concise, professional, and purposeful.

### 3.3 Forbidden Code Artifacts

| Artifact              | Rule                                                                              |
| --------------------- | --------------------------------------------------------------------------------- |
| Magic numbers         | Named constants for all tuning values and thresholds.                             |
| Force unwraps         | Prohibited unless at an unavoidable platform boundary, with inline justification. |
| `print()` / `NSLog()` | Prohibited in committed Swift code. Use structured logging behind debug flags.    |
| `print()` in Python   | Prohibited in committed code. Use `logging` or `structlog`.                       |
| Stringly-typed APIs   | Use enums, constants, or typed identifiers.                                       |
| Dead code             | If unused, delete it. No commented-out blocks. No `_unused` variables.            |
| Placeholder stubs     | No empty function bodies. No `fatalError("not implemented")`.                     |
| Em dashes in source   | Prohibited in all source files per S2.3.                                          |

### 3.4 Swift-Specific Standards

- Target: Swift 6.2, iOS 26+.
- Strict concurrency: `@MainActor` on all view models and state-owning types.
- `@Observable` macro (Observation framework) for state. No legacy `ObservableObject` / `@Published` /
  `@EnvironmentObject`.
- `async/await` for all asynchronous operations. No completion handler patterns unless wrapping a platform API that
  requires it.
- `Sendable` conformance where data crosses isolation boundaries.
- Access control: mark everything as restrictive as possible. `private` by default, `internal` only when required,
  `public` only for module boundaries.
- Liquid Glass suppressed. Opaque backgrounds, UIKit appearance overrides per design doc S6.

### 3.5 Python-Specific Standards

- Target: Python 3.13+.
- `mypy --strict` clean. Complete type annotations on all function signatures.
- `ruff format` compliant (line length 88, double quotes, space indent).
- `ruff check` clean against the rule set: F, E, W, I, UP, B, SIM, C4, PTH, S, A, EM, T20, RET, PERF, RUF.
- Google-style docstrings on public interfaces. No type duplication in docstrings.
- `pathlib.Path` exclusively. No `os.path`.
- Exception messages assigned to variables before `raise` (ruff EM).
- No hardcoded secrets. No `eval()`/`exec()` on untrusted input. No `shell=True` with dynamic arguments.

**Verification before declaring Python work complete:**

```bash
ruff format --check .
ruff check .
mypy --strict .
```

If any command fails, the task is not done.

## S4 Git Workflow

### 4.1 Branch Protection

**`main` is protected.** All changes MUST reach `main` through a pull request.

**Agent Refusal Policy:** If instructed to commit directly to `main` without explicit user override, the agent MUST:

1. **HALT** -- Do not commit.
2. **REJECT** -- State: "Direct commits to `main` are prohibited per S4.1."
3. **OFFER** -- Create the appropriate branch and open a PR instead.

### 4.2 Branch Naming

| Prefix             | Purpose                                    |
| ------------------ | ------------------------------------------ |
| `feat/<scope>`     | New functionality                          |
| `fix/<scope>`      | Bug fixes                                  |
| `refactor/<scope>` | Structural changes without behavior change |
| `chore/<scope>`    | Config, tooling, build settings            |
| `docs/<scope>`     | Documentation changes                      |

Scope MUST be a short, lowercase identifier for the affected area (e.g., `api`, `contract-book`, `admin`, `sync`,
`signatures`, `theme`).

Squash merge preferred. Delete remote branch after merge.

### 4.3 Commit Message Format

```
type(scope): imperative description
```

| Type       | Purpose                               |
| ---------- | ------------------------------------- |
| `feat`     | New functionality                     |
| `fix`      | Bug fix                               |
| `refactor` | Restructuring without behavior change |
| `test`     | Test additions or modifications       |
| `chore`    | Config, tooling, build settings       |
| `docs`     | Documentation changes                 |

**Rules:**

- Subject line: imperative mood, under 72 characters, no trailing period.
- Scope: required. Short identifier for the affected module.
- Body (optional): explain **why**, not **what**. The diff shows what changed.
- No vague commits: "fix stuff", "updates", "WIP", "misc changes" are rejected.
- No AI attribution in commit messages. Protocol Zero (S2) applies.

**Examples:**

```
feat(contract-book): add dynamic pagination for long articles
fix(sync): handle offline signature upload retry
refactor(theme): extract color tokens to dedicated constants
chore(scripts): add typographic lint scanner
docs(design-doc): update skill registry with new entries
```

### 4.4 Pull Request Requirements

Every PR MUST include:

| Section              | Content                                                              |
| -------------------- | -------------------------------------------------------------------- |
| **Title**            | Conventional Commits format, under 70 characters, imperative mood.   |
| **What changed**     | Concise summary of the modification.                                 |
| **Why it changed**   | Motivation, linked requirement, or design doc section reference.     |
| **Risk surface**     | What could break. Affected modules or services.                      |
| **Testing notes**    | How the change was verified. Steps to reproduce, device, OS version. |
| **Protocol Zero**    | `./scripts/protocol-zero.sh` exits 0 on the branch.                  |
| **Typographic Lint** | `./scripts/check-em-dashes.sh` exits 0 on the branch.                |

Empty PR descriptions are rejected. PRs without testing notes are rejected.

## S8 Dependency Approval Policy

### 8.1 No Automatic Installation

The agent MUST NOT install, add, or modify dependencies without explicit user approval. This applies to:

- Swift Package Manager dependencies.
- Python packages (`requirements.txt`).
- Any third-party framework, library, or SDK.
- Build tool plugins.
- HTMX or frontend library versions.

### 8.2 Approval Protocol

Before proposing a dependency, the agent MUST present:

| Field                       | Content                                                             |
| --------------------------- | ------------------------------------------------------------------- |
| **Package name**            | Exact identifier (e.g., SPM URL, PyPI name).                        |
| **Version**                 | Exact version to pin, not a range.                                  |
| **Purpose**                 | What capability gap it fills.                                       |
| **Alternatives considered** | Why a platform-native or standard library solution is insufficient. |
| **Scope of impact**         | Which modules depend on it (iOS, backend, or both).                 |

The user must explicitly approve before the agent proceeds.

### 8.3 Platform-Native Preference

Prefer Apple frameworks, Python standard library, and existing stack components. Third-party dependencies are justified
only by a clear capability gap that cannot be addressed with native APIs.

1. **Check the design document first.**
2. **Read the relevant source files.**
3. **If still uncertain, ask the user.**

### 9.3 No Assumed Context

- What the user intends when instructions are ambiguous.
- That a pattern seen in the iOS app applies to the backend or vice versa unless the design doc states it.
- That prior conversation context carries forward to a new task unless explicitly referenced.

## S10 Engineering Standards

### 10.1 General Principles

- **No magic numbers.** Named constants for all tuning values and thresholds.
- **No force unwraps or unchecked casts** unless at an unavoidable platform boundary with inline justification.
- **No `print()` debugging in committed code.** Use structured logging gated behind debug flags.
- **No stringly-typed APIs.** Use enums, constants, or typed identifiers.
- **No dead code.** If unused, delete it. No commented-out blocks.

### 10.2 File Discipline

- One primary concern per file. File name reflects its content.
- No empty, stub, or placeholder files.
- Prefer editing existing files over creating new ones. Every new file must be justified.
- No README, CONTRIBUTING, LICENSE, or CHANGELOG generation unless explicitly requested.

### 10.3 Architecture

- Clear separation between I/O boundaries (network, filesystem, user input) and core logic.
- Functions do one thing. Files contain one concept.
- Error handling at system boundaries. Internal code may trust its own invariants.
- No god files combining unrelated concerns.
- No circular dependencies between modules.
- No global mutable state.
- No over-engineering: no abstraction layers, factory patterns, or dependency injection frameworks beyond what the
  design doc specifies.

### 10.4 Security

**These rules are BLOCKING regardless of context.**

**Forbidden:**

- Hardcoded secrets, API keys, tokens, or credentials in source files.
- `.env` files committed to version control.
- Plaintext passwords in comments, logs, or test fixtures.
- `eval()` or equivalent dynamic code execution on untrusted input.
- Shell command construction from unvalidated strings.

**Mandatory:**

- Secrets loaded from environment variables or `.env` files (gitignored).
- All user-facing input validated at the boundary.
- HTTPS for all network calls. No HTTP fallbacks.
- API keys stored in iOS Keychain, never in source. Backend keys hashed with bcrypt/argon2.

**Constraints:**

- Under 70 characters total.
- Imperative mood. No trailing period.
- Scope required. Lowercase, hyphen-separated identifier for the affected area.
- Type must be one of: `feat`, `fix`, `refactor`, `test`, `chore`, `docs`.

**Rejected titles and reasons:**

| Title                | Violation                          |
| -------------------- | ---------------------------------- |
| `fix stuff`          | No scope, vague description.       |
| `updates`            | No type, no scope, no description. |
| `WIP: admin panel`   | WIP is not a valid type.           |
| `Fixed the sync bug` | Past tense, no scope delimiter.    |

### 16.2 Mandatory PR Sections

Every PR description MUST include:

| Section                  | Content                                                                             |
| ------------------------ | ----------------------------------------------------------------------------------- |
| **Summary**              | Concise explanation of intent. Not a restatement of the diff.                       |
| **Design Doc Alignment** | Reference to the design doc section(s) implemented. Deviations declared explicitly. |
| **Risk Surface**         | Failure modes and affected components described.                                    |
| **Testing**              | Device, OS version, scenarios verified, tests added.                                |
| **Protocol Zero**        | `./scripts/protocol-zero.sh` exits 0 on the branch.                                 |
| **Typographic Lint**     | `./scripts/check-em-dashes.sh` exits 0 on the branch.                               |

Empty PR descriptions are rejected. PRs without testing notes are rejected.

### 16.3 Squash Merge Governance

This repository uses squash merge (S4.2). In a squash merge workflow, the PR title becomes the commit message on the
`main` branch. This makes the PR title the authoritative historical record.

Every constraint that applies to commit messages (S4.3) applies equally to PR titles.

## S17 Memory System

**Priority: BLOCKING. This section governs persistent context across sessions.**

### 17.1 Memory Location

Persistent memory resides at the Claude Code auto memory path:

```
~/.claude/projects/-Users-Dinesh-Desktop-exhibitA/memory/
```

`MEMORY.md` is auto-loaded into every session context. Additional topic files may be created in the same directory and
linked from MEMORY.md.

### 17.2 Mandatory Update Triggers

The agent MUST update persistent memory at each of the following events:

| Trigger                          | Action                                                                                          |
| -------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Session start**                | Read MEMORY.md and linked topic files. Clear any Next Session notes after reading.              |
| **After major implementations**  | Record what was built, patterns chosen, and new modules in the appropriate memory section.       |
| **After architectural decisions** | Append a dated entry to the decisions section. Format: `- YYYY-MM-DD: <summary> -- <rationale>` |
| **After git pushes**             | Record branch, PR number, and summary. Update in-flight tracking.                               |
| **Session end / /save-memory**   | Full sync: decisions, progress, stack changes, and Next Session handoff note.                   |

### 17.3 Write Rules

- MEMORY.md must stay under 200 lines. Move detail to topic files.
- Decision entries are append-only. Never delete or rewrite past decisions.
- Completed items are append-only with dates. Never delete.
- In-flight and Next Session sections are rewritten to reflect current state.
- Use the Edit tool to update in place. Do not recreate files.
- Do not duplicate information already in this CLAUDE.md or the design doc.

### 17.4 Skill and Command

- **Skill:** `memory-manager` (`.claude/skills/memory-manager/SKILL.md`) -- encapsulates all read/write logic.
- **Slash command:** `/save-memory` -- triggers a full session-end memory sync.
